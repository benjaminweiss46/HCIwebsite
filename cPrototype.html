Design Evolution

Slow Chat

The slow chat button was a feature that allowed to slow down the chat recieved by zoom. This is done so the host never misses a comment at the expense of a potential delay on the hosts chat. While this option was good in practice if confused users in testing who didn't understand what this button did. Even with some explination some users didn't get what it really did. So this was completely removed.

Tool Tips

While we may have believed our interface was so easy to use it needed no explanation. This proved to be incorrect, some user had no idea what to do. To combat this we have included tooltips on some of the buttons to indicate what actions the button performs.

More Options

While we wanted have more than 4 graph options, and give users maximal optionality we didn't give them a way to get more in practice. Therefore we added a button to the graph options so that users can select from a much larger range of options.

Graph Customization

We initially expected users to customize the graphs by clicking on graph elements. Not many users understood this interation. Therefore we moved graph customization to its own section, with labels and input boxes so users know exactly how to customize the graph.

Preview Selection

We expected users to understand that to see their graph in a larger window and edit it they can click on the preview. This was a poor assumption and testing proved users didn't do this. Therefore we added an explicit select button to select a certain visualization.

User Poputation

In the project proposal, we described 3 distinct personas who could gain benefit by our product, a 51 year old CEO, a university professor, and an engineering student. All of the 3 personas are unsatisfied with the current Zoom service, and are willing to experience products that will improve their experience. However, we decided to apply the test plan of the computer prototype to 2 of them because of their experience and knowledge. The user population for our application need to have basic knowledge of java and command lines.

The first is Robert Woods, age 43, a university professor, and the second is Christopher Lee, age 21, a student of McGill Engineering. These users are able to configure their environment and install the application with barely any difficulties, because they have the basic knowledge of computers. They are also comfortable with trying out new functions that are provided with our application. By identifying these users more easily, the test results can be analyzed while considering users to make corresponding improvements to the prototype.
